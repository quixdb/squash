1 # Building Squash
2 
3 Note: [Squash currently does not support
4 Windows](https://github.com/quixdb/squash/issues/86). It should work
5 on at least Linux, BSD, and OS X. If you encounter problems please
6 file an issue.
7 
8 ## Dependencies
9 
10 In order to build Squash, you'll need CMake, make, a C compiler, a C++
11 compiler, and [Ragel](http://www.colm.net/open-source/ragel/).
12 Additionally, if you want to build the tests (which is a good idea)
13 you'll need glib. The necessary packages vary by distribution, but
14 for some of the more popular distributions:
15 
16 * **Debian/Ubuntu** — gcc g++ ragel cmake make libglib2.0-dev
17 * **Fedora/RHEL/CentOS** — gcc gcc-c++ ragel cmake make glib2-devel
18 
19 Squash includes copies of all the libraries it uses for
20 compression/decompression. That said, you may prefer to use system
21 copies when available. By default, if a system library is installed
22 when configuring the build Squash will use it, if not it will fall
23 back on its internal copy. Again, the package names vary by
24 distribution, but for some of the more popular distributions:
25 
26 * **Debian/Ubuntu** — libbz2-dev liblzma-dev liblzo2-dev libsnappy-dev
27  zlib1g-dev
28 * **Fedora/RHEL/CentOS** — bzip2-devel xz-devel liblzf-devel lzo-devel
29  snappy-devel zlib-devel

31 ## Building from git
32 
33 If you are building from a release, skip this section.
34 
35 Squash makes very heavy use of git submodules to pull in code from
36 other projects, mostly for the libraries used by each plugin. When
37 possible Squash will use system libraries instead of its own copy, but
38 only a relatively small number of plugins use libraries which are
39 typically packaged by distributions.
40 
41 If you are on a UNIX-like system with bash, you can simply call the
42 `autogen.sh` script (located in the top level of the Squash sources)
43 just as you would call the `configure` script. If not you can
44 update the submodules manually—from the top level of the Squash
45 sources:
46 
47 ~~~{.sh}
48 git submodule update --init --recursive
49 ~~~
50 
51 `autogen.sh` simply invokes this command then calls `configure` with
52 the arguments you passed to it.
53 
54 Once you have all the submodules your source tree should be in
55 basically the same state as a release (except you still have all the
56 git data), so proceed to the next section.
57 
58 ## Building from a release
59 
60 If you are on a UNIX-like system (basically non-Windows) with bash
61 (basically non-BSD), simply run the `configure` script with the same
62 arguments you would pass to the `configure` script of any
63 autotools-based project:
64 
65 ~~~{.sh}
66 ./configure
67 ~~~
68 
69 This will translate the arugments you pass to the CMake versions and
70 invoke CMake. For a list of supported arguments, pass `--help`.
71 
72 If you don't have bash, you'll have to call CMake manually:
73 
74 ~~~{.sh}
75 cmake .
76 ~~~
77 
78 At this point, all you need to do is call `make`, and likely `make
79 install`, just as you would for any other project. If you encounter
80 an error at this point please file a bug—all misconfigurations should
81 be detected by `configure`/`cmake`.
